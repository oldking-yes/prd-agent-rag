# PRD 编写方法论与最佳实践

## 什么是 PRD？

产品需求文档（Product Requirements Document，PRD）是产品经理的核心交付物，用于在团队中传递产品决策、对齐期望、定义交付范围。

一份优秀的 PRD 不是"需求清单"，而是**决策记录 + 交付契约**。

## 现代 PRD 的核心原则

### 1. 从问题出发，而不是从方案出发
- 用数据、用户反馈、支持工单来证明问题存在
- 在描述功能之前，先定义要解决的问题
- **反面例子**："我们要做一个分享功能"
- **正面例子**："用户调研显示 67% 的用户需要把报表分享给团队，当前只能截图"

### 2. 先定义成功，再定义功能
- 每个 PRD 必须有可衡量的成功指标
- 指标必须包含基线和目标值
- 例："激活率从 34% 提升到 55%"

### 3. 明确不做什么
- "非目标"（Non-Goals）比"目标"更重要
- 明确排除范围，防止 scope creep
- 例："本版本不做社交功能"

### 4. 保持文档生命力
- PRD 不是写完了就归档的
- 随着调研深入持续更新
- 过时的 PRD 比没有更糟糕

## PRD 的两种形态

### 轻量版（1-2 页）
适用场景：单团队功能、现有流程改进、6 周以内的项目
包含：问题陈述、目标指标、范围、验收标准

### 完整版（4-8 页）
适用场景：跨团队项目、新产品模块、高风险/高可见度功能
包含：全部十个标准章节

## 标准 PRD 结构

| # | 章节 | 内容要点 |
|---|------|---------|
| 1 | **问题陈述** | 用数据证明的痛点，非主观意见 |
| 2 | **目标与指标** | 可衡量的结果，含基线和目标值 |
| 3 | **范围/非目标** | 明确包含什么、不包含什么 |
| 4 | **用户画像与故事** | "作为[用户]，我想要[功能]，以便[收益]" |
| 5 | **功能需求** | 用 MoSCoW 或 P0/P1/P2 排序 |
| 6 | **验收标准** | Given/When/Then 格式，覆盖边界情况 |
| 7 | **约束与依赖** | 技术、法律、资源、时间限制 |
| 8 | **开放问题** | 还未确定的决策，标注负责人和截止日期 |
| 9 | **发布计划** | 灰度策略、功能开关、回滚方案 |
| 10 | **风险与应对** | 可能出问题的地方 + 应对方案 |

## 验收标准的最佳写法

使用 Given/When/Then 格式：

```
Given [前提条件]
When [触发动作]
Then [预期结果]
```

示例：
```
Given 用户已登录且未购买会员
When 用户点击"生成完整 PRD"按钮
Then 弹出引导开通会员的对话框
```

## 常见错误与避坑

| 错误 | 正确做法 |
|------|---------|
| 闭门造车 | 早期就让工程、设计参与评审 |
| 15 页以上的长篇大论 | 默认一页纸，详情外链 |
| 功能没有上下文 | 每个功能点都关联用户问题 |
| 把 PRD 当最终文件 | 随知识更新持续迭代 |
| 遗漏"非目标" | 明确写出不做的事项 |
| 验收标准模糊 | 使用 Given/When/Then 格式 |
| 没有开放问题区 | 记录未定事项及负责人和期限 |

## 关键检查清单

撰写 PRD 后自检：
- [ ] 问题是否用数据支撑？
- [ ] 成功指标是否可衡量？
- [ ] 目标用户是否明确？
- [ ] 功能优先级是否清晰？
- [ ] 验收标准是否可测试？
- [ ] 是否列出了"不做"的事项？
- [ ] 依赖项是否已确认？
- [ ] 是否经过工程和设计的评审？
- [ ] 开放问题是否有负责人和截止日期？
