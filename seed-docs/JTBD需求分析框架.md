# Jobs To Be Done (JTBD) 框架——产品需求分析的核心方法论

## 什么是 JTBD？

JTBD（Jobs To Be Done）是一个产品管理方法论，由哈佛商学院的 Clayton Christensen 教授和 Tony Ulwick（Strategyn）共同创立。它的核心洞察是：

> **人们不是购买产品，而是"雇佣"产品来帮他们在特定情境下取得进展。**

这个框架把分析焦点从"用户是谁"（ demographics ）转移到"用户想完成什么任务"。

---

## 为什么 JTBD 对 PRD 很重要？

传统的 PRD 经常掉进"功能列表"的陷阱——列出要做什么，但不解释为什么。

JTBD 提供了回答"为什么"的工具：
- 用户真正的动机是什么？
- 在什么场景下用户需要这个产品？
- 用户现在用什么替代方案？
- 怎样的体验才算"完成工作"？

---

## JTBD 陈述模板

Facebook/Instagram 等产品团队广泛使用的格式：

> **当我** [情境/场景]…
> **但是** [障碍/困难]…
> **帮我** [目标/任务]…
> **以便** [结果/目的]…

### 示例：AI 刷题小程序

| 元素 | 内容 |
|------|------|
| **当我** | 想要准备面试但不知道从何开始 |
| **但是** | 找不到针对性的题库，刷题枯燥难坚持 |
| **帮我** | 智能推荐适合我水平的题目，提供即时解析 |
| **以便** | 高效利用碎片时间准备面试 |

## 六种"工作"类型

| 类型 | 说明 |
|------|------|
| **核心功能工作** | 用户试图完成的主要任务 |
| **期望结果** | 用户想要达成的理想结果 |
| **相关工作** | 次要目标 |
| **情感与社交工作** | 用户想如何被感受和被看待 |
| **消费链工作** | 学习、支持、购买等环节 |
| **财务期望** | 预算/ROI 考量 |

---

## JTBD 实施十步法

1. **获得领导层支持** —— 需要转变思维方式
2. **组建跨职能团队** —— 产品、设计、工程、运营、市场
3. **开展用户访谈** —— 理解动机，而不仅仅是行为
4. **以"工作"为优先级指南** —— 每个功能都关联具体工作
5. **对齐"工作陈述"与用户细分**
6. **绘制"工作旅程"** —— 定义、定位、准备、确认、执行、监控、修改、结束
7. **嵌入工作流** —— 在用户故事中包含"工作陈述"
8. **重新思考指标** —— 衡量节省的时间、消除的痛点
9. **培训团队** —— workshop 和技能建设
10. **持续迭代** —— 定期重新审视"工作陈述"

## JTBD 与用户故事的对比

| 维度 | JTBD | 用户故事 |
|------|------|---------|
| **关注点** | 期望结果 | 具体方案 |
| **模板** | 当…但是…帮我…以便… | 作为…我想要…以便… |
| **阶段** | 产品发现 | 开发执行 |
| **目的** | 理解"为什么" | 定义"做什么" |

## 真实案例

### Cordis 医疗器械
- **应用 JTBD 前**：仅 1% 市场份额
- **JTBD 洞察**：医生需要的不是"更好的球囊"，而是"提高手术成功率、降低风险"
- **结果**：市场份额增长到 20%+，19 款新产品成为畅销品

### 微软 Software Assurance
- **问题**：客户看不到该计划的价值
- **JTBD 洞察**：客户真正想"优化 IT 预算"、"管理风险"、"减少业务中断"
- **结果**：年收入同比增长 100%

## 在 PRD 中使用 JTBD

在 PRD 的"问题陈述"和"用户故事"章节中融入 JTBD：

### 传统写法
```
用户需要一个刷题功能。
```

### 融入 JTBD 的写法
```
用户调研发现：
- 情境：准备面试的候选人每天通勤时间 45 分钟
- 挣扎：现有题库要么太简单要么太难，无法针对性练习
- 需要的工作：在碎片时间高效刷题，即时获得反馈
- 期望结果：面试前准备效率提升 2 倍
```
